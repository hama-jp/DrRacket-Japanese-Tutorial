## 問題 1. `mathlib/main.rkt` に `cube` と `hypot3` を追加

### ヒント

本章の既存 `square` / `hypot` と同じ書き方。`hypot3` は `hypot` の 3 次元版で、内部で `square` を再利用できます。

### 部分解答

```racket
(define (cube x) (* x x x))

(define (hypot3 a b c)
  (sqrt (+ (square a) (square b) (square c))))
```

### 完全解答

完全版は [`solutions/ch11/mathlib/main.rkt`](ch11/mathlib/main.rkt) を参照してください。要点:

```racket
(provide square hypot cube hypot3)

(define (square x) (* x x))
(define (hypot  a b)   (sqrt (+ (square a) (square b))))
(define (cube   x)     (* x x x))
(define (hypot3 a b c) (sqrt (+ (square a) (square b) (square c))))
```

`user.rkt` から呼ぶとこうなります。

```bash
$ racket solutions/ch11/user.rkt
49
5
64
7.0
```

`hypot3 2 3 6` は 2² + 3² + 6² = 49、その平方根は 7。整数演算だけで足し合わせたあと `sqrt` が呼ばれるため、結果は **不正確数** (`7.0`) で返ります。厳密値が欲しければ `exact->inexact` 抜きで `(integer-sqrt (+ 4 9 36))` を使います。

## 問題 2. `module+ test` を追加して `raco test` で走らせる

### ヒント

同じファイルの末尾に `(module+ test ...)` を書くだけ。`rackunit` を中で `require` する。

### 部分解答

```racket
(module+ test
  (require rackunit)
  (check-equal? (cube 3) 27)
  (check-equal? (hypot3 2 3 6) 7))
```

`check-equal?` は **厳密値** の比較。`hypot3` は `sqrt` が絡んで不正確数になるので、誤差込みで比較する場合は `check-=` を使います。

### 完全解答

完全版は [`solutions/ch11/mathlib/main.rkt`](ch11/mathlib/main.rkt) に同梱。実行:

```bash
$ raco test solutions/ch11/mathlib/main.rkt
raco test: (submod "solutions/ch11/mathlib/main.rkt" test)
7 tests passed
```

テストが 1 つでも失敗すると、メッセージに該当行番号と期待値 / 実際値が出ます。`rackunit` の雑なルール:

| 書き方 | 何を比べる |
| --- | --- |
| `(check-equal? a b)` | `equal?` |
| `(check-eq?    a b)` | `eq?` |
| `(check-= a b ε)` | 浮動小数の誤差許容 |
| `(check-true  x)` | `x` が真 |
| `(check-false x)` | `x` が `#f` |
| `(check-exn pred thunk)` | `thunk` が `pred` にマッチする例外を投げる |

`module+ test` のうれしさは、テストを **本体コードと同じファイル** に置きつつ、`raco test` 時にしか読み込まれない点です。普通に `require` したときの起動コストに影響しません。

## 問題 3. 循環依存を作ってみる

### ヒント

本当に 2 つのファイルを作って、互いに `require` させるのが一番勉強になります。Racket は「実行前」に `require` 解決の段階で気づきます。

### 部分解答

```racket
;; a.rkt
#lang racket
(require "b.rkt")
(provide foo)
(define (foo x) (bar x))
```

```racket
;; b.rkt
#lang racket
(require "a.rkt")
(provide bar)
(define (bar x) (foo x))
```

### 完全解答

添付コード: [solutions/ch11/circular/a.rkt](ch11/circular/a.rkt) / [b.rkt](ch11/circular/b.rkt)

実行:

```bash
$ racket solutions/ch11/circular/a.rkt
standard-module-name-resolver: cycle in loading
  at path: .../solutions/ch11/circular/a.rkt
  paths:
   .../solutions/ch11/circular/a.rkt
   .../solutions/ch11/circular/b.rkt
```

**回避策** の言語化:

1. **片方向の依存に直す**。普通は「上位 → 下位」「アプリ → ライブラリ」と一方向にする。
2. **共通部分を 3 つ目のモジュール (`common.rkt`) に切り出す**。`a.rkt` と `b.rkt` が共に `common.rkt` を `require` する構成にすれば、直接の循環は消える。
3. **依存の向きを逆転する**。`b.rkt` が `a.rkt` の関数を直接呼ぶ代わりに、関数をコールバックとして受け取るインタフェース(`provide` する関数に高階関数を渡す)にする。
4. **データと振る舞いを分ける**。両者が共有しているのがデータ型だけなら、データ型定義を別ファイルに切り出す。

雑なまとめ: **「依存関係は DAG(有向非巡回グラフ)になるように設計する」** のが原則。循環を発見したら、責務をもう一段切り直すサインと受け取るとよいです。

添付の [`common.rkt`](ch11/circular/common.rkt) は 2. の切り出しの最小例です。
