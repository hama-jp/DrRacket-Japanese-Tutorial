## 問題 1. `my-and` — 短絡評価の `and` をマクロで

### ヒント

問題文のテンプレートそのものが骨子。3 つの節を書き分ける:

- 引数ゼロ → `#t`
- 引数 1 つ → **評価して** 返す(残りがないので打ち切り)
- 2 個以上 → 先頭を `if` で真偽を見て、真なら残りを再帰的に `my-and`、偽なら `#f`

### 部分解答

```racket
(define-syntax my-and
  (syntax-rules ()
    [(_)          #t]
    [(_ e)        e]
    [(_ e1 e2 ...) (if e1 (my-and e2 ...) #f)]))
```

### 完全解答

添付の [`solutions/ch16/macros.rkt`](ch16/macros.rkt) にテスト付きで入れてあります。確認:

```bash
$ raco test solutions/ch16/macros.rkt
raco test: (submod "solutions/ch16/macros.rkt" test)
N tests passed
```

重要なのは **短絡評価** が本当に効いているかのテストです。添付では `parameter` を使って「`#f` の後ろの式は評価されない」ことを確かめています。

```racket
(define hit (make-parameter 0))
(my-and #f (begin (hit (+ (hit) 1)) 'unused))
(hit) ; => 0 でなければならない
```

`and` / `or` のマクロ版は「動く」だけでなく「**短絡** する」ところまで含めて正しい実装です。

## 問題 2. `unless-let`

### ヒント

「値を 1 度だけ束縛して、その値が偽なら本体を実行」 。普通の `let` + `unless` の合成マクロです。

### 部分解答

```racket
(define-syntax unless-let
  (syntax-rules ()
    [(_ name expr body ...)
     (let ([name expr])
       (unless name body ...))]))
```

### 完全解答

```racket
(define-syntax unless-let
  (syntax-rules ()
    [(_ name expr body ...)
     (let ([name expr])
       (unless name body ...))]))

(unless-let x (read-line)
  (displayln "no input"))
```

確認したいポイントが 2 つあります。

1. **`expr` が 1 回しか評価されない** こと。マクロ展開で `name` に直接 `expr` が埋め込まれてしまうと、`unless` 側と本体側で 2 回評価される事故が起きます。上の書き方なら `let` が吸収してくれて安全。
2. **`name` が本体から見える** こと。`unless` の本体は `let` の内側なので、`name` がスコープに入ります。

添付テストでは、副作用が 1 度しか起きないことを `evals` で確認しています:

```racket
(define evals 0)
(define (side-effect!) (set! evals (+ evals 1)) 42)
(unless-let v (side-effect!) (void))
evals ; => 1
```

`syntax-rules` は **衛生的なマクロ**(hygienic macro)なので、同じ名前の識別子が衝突する事故も自動で防がれています。これは C プリプロセッサの `#define` と大きく異なる Scheme / Racket 流の利点です。

## 問題 3. mini-lisp に `cond` をマクロ展開する戦略

### ヒント

ホスト側(Racket)の `cond` と同じように、**mini-lisp 内** でも `cond` を `if` の入れ子に書き換える「マクロ展開」が理論的には可能です。ただし第 14 章の mini-lisp には `define-syntax-rule` 相当の仕組みが入っていないので、**実装する** のではなく **設計を言語化** するのが本問の狙い。

### 部分解答 — 考える順番

1. **展開関数**: `(cond ...)` を受け取って `if` の入れ子に変える純粋関数 `cond->if` を書く。たとえば

   ```text
   (cond [c1 body1] [c2 body2] [else body3])
   ↓
   (if c1 body1
       (if c2 body2
           body3))
   ```

2. **展開タイミング**: `mini-eval` が `cond` を見たときに、**評価する前に** `cond->if` を呼んで変換する。つまり `cond` は「評価規則の節」ではなく「**評価前の書き換え**」として扱う。

3. **ユーザマクロの仕組み**: 本格的には `(define-syntax name (lambda (expr) ...))` のようなフォームを導入して、ユーザが自分でマクロを書けるようにする。`mini-eval` の先頭でマクロ展開ステップを挟み、マクロに該当したら展開結果を再度評価する。

### 完全解答 — 書き換え関数の例

```racket
(define (cond->if clauses)
  (cond
    [(null? clauses) '(void)]
    [(eq? (caar clauses) 'else)
     `(begin ,@(cdar clauses))]
    [else
     `(if ,(caar clauses)
          (begin ,@(cdar clauses))
          ,(cond->if (cdr clauses)))]))
```

これを `mini-eval` の `cond` 節で呼ぶと、

```racket
[(list* 'cond clauses)
 (mini-eval (cond->if clauses) e)]
```

のように「`cond` は `if` と `begin` の合成にすぎない」という構造が、**コードそのもので** 見える形になります。

発展として、`define-syntax` を mini-lisp に入れる最小の形:

```racket
(struct macro (transformer))

;; define-syntax name (lambda (stx) ...) を登録する節
[(list 'define-syntax name (list 'lambda (list stx) body ...))
 (env-set-global! e name
                  (macro
                   (lambda (form) (mini-eval `(let ([,stx ',form]) ,@body) e))))]

;; mini-eval の先頭にマクロ展開ステップを挟む
[(and (pair? expr)
      (macro? (and (symbol? (car expr))
                   (env-lookup-safe e (car expr)))))
 (mini-eval ((macro-transformer
              (env-lookup e (car expr)))
             expr)
            e)]
```

これは骨子だけですが、`cond`, `when`, `unless`, `let*` などを **全部マクロに追い出す** という発想に繋がります。Scheme / Racket の文化では「言語のコア + マクロでライブラリ」というスタイルが強く、この設計思想の原型を mini-lisp の拡張で体感できます。

第 17 章の継続や遅延評価の話を読んだ後に戻ってくると、同じ流れで「`delay` / `force` もマクロで書ける」「`for` ループもマクロで書ける」という見方が自然にしっくりきます。
